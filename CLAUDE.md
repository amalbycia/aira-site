## Conventions

**Naming**
- Files: `kebab-case.ts` for lib/utils, `PascalCase.tsx` for React components
- Exports: named exports for utilities, default exports for page/layout components

**Component structure — mobile/desktop isolation**
- If a component has meaningfully different layout or animation on mobile vs. desktop,
  split it: `components/desktop/HeroSection.tsx` and `components/mobile/HeroSection.tsx`
- The parent page imports the correct one via a responsive wrapper or conditional render
- Editing one file must never touch the other

**GSAP matchMedia pattern**
- Always use the `createMatchMedia()` helper from `lib/gsap.ts` inside `useGSAP()`
- Put desktop animations inside `mm.add("isDesktop", () => { ... })`
- Put mobile animations inside `mm.add("isMobile", () => { ... })`
- Call `mm.revert()` in the `useGSAP` cleanup return
- Never put shared logic inside a condition block — hoist it above the `mm.add()` calls

**Sanity**
- Always use `defineField` / `defineType` — never raw objects in schemas
- Field `description` strings are written for the client (non-technical language)
- The `siteSettings` type is a singleton — only one document should ever exist

**Images**
- Use `urlFor()` from `lib/imageUrl.ts` for all Sanity images
- Use `getBunnyCdnUrl()` from `lib/bunny.ts` for all Bunny Storage assets
- Always provide `alt` text

**Phone optimization (check at every step) — STRICT, see [MOBILE-RULES.md](./MOBILE-RULES.md)**
- Every component or style change must be checked against mobile viewport widths, not just desktop
- Follow [MOBILE-RULES.md](./MOBILE-RULES.md) exactly — touch target sizes, font minimums, breakpoint logic, overflow handling
- Use fluid units (`clamp()`, `em`, the Osmo scaling system) so text and spacing scale down correctly — never assume desktop sizing reads fine on phone
- Buttons, nav, and text must never overflow, wrap awkwardly, or get clipped on small screens
- Test at 320px, 375px, and 768px widths minimum before calling any UI change done
- Treat "looks right on phone" as part of done, not a follow-up pass

## UI/UX Design Guidelines

A set of principled constraints for building interfaces that are intentional, accessible, and free from common anti-patterns.

### 1. Visual Identity & Branding

**Avoid "AI Slop" Defaults**
Never use generic purple gradients, massive drop shadows, default AI-generated typography, or standard AI icons (like the ubiquitous sparkle/magic wand). Do not use system emojis as a substitute for proper iconography, and ensure all image assets are high-resolution — avoid blurry placeholders.

**Strict Design Tokens**
Establish a foundational set of design tokens — spacing, typography, border radii, colors — and adhere to them throughout. Rely on the specific brand color palette; do not improvise.

**Prevent "Frankenstein" UI**
Ensure visual continuity across all pages and sections. Do not mix different visual languages (e.g., raw markdown outputs alongside styled UI, or flat and 3D elements used inconsistently in the same context).

### 2. Layout and Information Hierarchy

**Eliminate Purposeless Elements**
Do not place gray boxes, dividers, or UI chrome on screen without a clear functional reason. Every visible element must earn its place.

**One Focus per Scroll Depth**
At every scroll position, there should be only one primary CTA or priority message. Competing calls to action dilute intent and confuse users.

**Clean Component Alignment**
Preview images, cards, and screenshots must be consistently sized and correctly aligned. Avoid the "boxes in boxes" look — redundant nested borders create visual noise without hierarchy.

**Viewport Sizing**
Do not make the hero section take up exactly 100% of the vertical space without visual cues. Ensure elements from the next section "peek" above the fold so users intuitively know there is more content below.

**Hero Section Clarity**
The H1 and immediate subtext must instantly communicate what the product is, who it is for, and why they should care — accompanied by a clear call-to-action.

**Constrain Text Styles**
Limit typography styles. Do not invent 4 or 5 different text sizes, colors, and label treatments for a single section. Keep hierarchy tight and readable.

**Contrast is Non-Negotiable**
Ensure absolute text legibility at all times. If text sits over a dynamic background or video, it must remain highly contrasted and readable. Never place bright navigation over auto-playing video.

### 3. Animation and Motion

**No Motion While Reading**
Never use background animations, moving lines, or shooting elements when the user is stationary and trying to read. Animation must not compete with content.

**Purposeful Motion Only**
Only use animations if they reinforce the brand or directly explain a product feature. Do not animate just because it is technically easy to do so.

**Respect the Scroll**
Never implement scrolljacking or hijack native browser scrolling to force users through JavaScript animations. If a user scrolls quickly, content must not fail to load or be missed.

### 4. Interactivity & Hover States

**Smooth, Standard Click Cues**
Hover effects must transition smoothly — no abrupt snapping. Indicate clickability through standard conventions: a hand cursor, a slight glow, or a one-shade shift. Do not use hover states that accidentally shift the surrounding layout.

**Never Hide Information**
Never hide core functionality, critical navigation, or essential text behind a hover state. This is a severe anti-pattern that is completely inaccessible on mobile devices.

**No Disappearing Acts**
Hovering over an element should never cause it to fade out, jump around, or displace surrounding content.

**Always Tooltip Icon-Only Buttons**
Any button that displays only an icon must include a tooltip — never assume the icon's meaning is universally understood.

### Anti-Pattern Reference

| Category | Pattern | Description |
|---|---|---|
| Color | Ubiquitous purple gradients | Identical purple/pinkish-purple gradients used across countless sites |
| Color | Generic emojis and icons | Standard system emojis or off-the-shelf icons in place of custom iconography |
| Color | Blurry assets | Low-resolution images used as video poster frames or hero visuals |
| Layout | 3×2 "bento box" layout | Features in a uniform grid — icon top, text below — repeated without variation |
| Layout | 100% viewport hero sections | Hero sized to fill the screen exactly, nothing peeking below the fold |
| Layout | Boxes in boxes | Cards with redundant nested borders that add noise without hierarchy |
| Animation | Scrolljacking | Native scroll hijacked for JS animations |
| Animation | Aggressive scroll-triggered fade-ins | Content fades in on a timer; fast scrollers miss sections entirely |
| Animation | Shooting meteors / cursor lights | Decorative background animations unrelated to content |
| Hover | Fade-out hover effects | Hover causes elements to disappear rather than invite a click |
| Hover | Layout-shifting hovers | Elements shift position on hover, displacing surrounding content |
| Hover | Information hidden behind hovers | Critical copy only visible on hover — inaccessible on touch devices |
| Hover | Hallucinated click targets | Elements styled as buttons that do nothing when clicked |
